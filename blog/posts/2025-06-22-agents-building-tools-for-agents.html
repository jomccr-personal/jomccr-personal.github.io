The Rise of Niche Software

June 22, 2025

Tags: agents, software engineering
Audience: technical

With the advent of agentic coding assistants like VS Code's agent mode, totally new IDEs like Cursor, and my personal favorite: the minimalist Claude code - some software developers have been left scratching their heads trying to figure out what exactly these new assistants are good at. This article will discuss three things to look out for and how to fix them to get the most out of building with agents. The truth is that this used to be true. Writing throwaway software was a luxury, something that rarely was done because the process was often many times more complicated than just doing the task and getting on with your life. Only the most major annoyances got the special treatment of automation at the behest of product management. Often each developer would build their own half-baked smattering of bash scripts and unholy makefile targets combined together into a lovecraftian monster that nobody fully understood.

However, after the introduction of agents, this all changed.

Throwaway code is now cheaper than ever to write (or generate). Often it can be done in a single prompt. What used to take a week to build properly now takes hours to get off the ground. This rapid development cycle for small developer experience focused utilities is the most underrated tool for increasing your productivity when working with agents.. If there is a manual interaction with something on the web, especially if it already has an existing API, you should consider building a command line tool that an agent can use for interacting with it if it doesn't exist already. The biggest boost from getting to use an agent vs a traditional chat-only LLM is, well, the agency that it has to call tools.

Take this example from Anthropic about the already popular gh command:

Claude knows how to use the gh CLI to interact with GitHub for creating issues, opening pull requests, reading comments, and more. Without gh installed, Claude can still use the GitHub API or MCP server (if you have it installed).

You should be thinking about how to build the gh for your team or service. Building niche tools for your agents can be very rewarding. I think it will turn out to be some of the most productive work that we are all totally skipping today. There will be a suite of these kinds of tools, simple bash tools for interacting with common services, that bubble to the top. Different workflows require different tools, and for my data engineering day job I've built afcli to make it easier to interact with development servers that you've stood up during development.

Why not use the existing airflow CLI?

It's niche enough that nobody has built it. The main feature that I had thought was missing from the airflow CLI was the --host functionality. I wanted a way to point to a running airflow host and skip having to find the docker container id, then run docker exec with the proper airflow command, etc. All of this indirection adds potential edge cases for the agent loop to break down. The legacy workflow

Before agents, developers could mount their code in a docker volume that would update regularly. The flow was:

  • Write code
  • Import error
  • Fix code
  • Trigger dag
  • Task failed
  • Fix code
  • Clear tasks
  • Finally watch logs succeed

This iteration loop was completely broken. Imagine a random slack huddle pop up throwing you out of the zone in the middle of this loop. Experienced developers have consistently noted that context switching is the number one killer of your productivity. The context switch between your agent and your system that you're working on is just as expensive.

Practical example with afcli

In your CLAUDE.md or equivalent, you'll want to add something like this: 

When building with airflow, check the dag and task level statuses with `afcli`.
Use `afcli` for fetching logs and to validate your changes.
After making a change, run `afcli` to kick off the dag or task that you're working on and iterate until every task succeeds.
Run `afcli --help` to understand common usage patterns and specific CLI options.

This may seem simple, but it's insanely effective. Giving your agent the instructions on how to iterate faster unlocks a tight loop that will let you turn features around faster than ever.
